# HybX Development System Design Document
## Hybrid RobotiX

---

## 1. Overview

The HybX Development System is a portable, repeatable development environment for the Arduino UNO Q and similar boards. It replaces Arduino App Lab entirely with a set of versioned Python bin commands that run over SSH, a bootstrap script for provisioning new boards, and a VSCode extension that integrates all commands into the editor.

**Guiding philosophy:** clean architecture, zero vendor lock-in, everything versioned.

*"I. WILL. NEVER. GIVE. UP. OR. SURRENDER."*

---

## 2. Repository Structure

```
HybX-Development-System/
  bin/          — Versioned Python bin commands and shared modules
  docs/         — Design documents, inventory, known issues
  scripts/      — Bootstrap script (newrepo.bash template)
  README.md     — Quick start and reference
```

Shared modules in `bin/` are unversioned — they are imported by versioned
commands and updated in place. Versioned commands are never modified after
creation; new versions are new files. Shared modules are the exception:
they evolve alongside the commands that use them.

The companion robot app repository is [hybotix/UNO-Q](https://github.com/hybotix/UNO-Q).

---

## 3. Design Principles

- **All Python, no bash or shell scripts** — every bin command is a versioned Python script
- **Versioned filenames** — `command-vX.Y.Z.py` with unversioned symlinks pointing to the latest
- **Configuration at the top** — all user-configurable variables live at the top of each script
- **Last-app memory** — commands that need an app name fall back to `~/.last_app` if none is provided
- **Install from source** — libraries install into `/usr/local`, never system packages
- **JSON for config** — configuration stored in JSON where applicable
- **Single source of truth** — the repo is authoritative; the UNO Q environment is always reproducible from it
- **Shared code lives in shared modules** — any logic used by two or more commands must be extracted into an importable module in `bin/`. Never duplicate code across command files. Current shared modules:
  - `hybx_config.py` — board config, library registry load/save, shared path constants
  - `libs_helpers.py` — library filesystem scanning, arduino-cli wrappers, sync logic

## 4. Bin Commands

All commands live in `bin/` as versioned Python scripts. Unversioned symlinks in `~/bin/` on the UNO Q point to the latest version of each command.

### 4.1 Command Reference

| Command | Latest | Description |
|---------|--------|-------------|
| `start` | v0.0.10 | Nuke Docker, clear cache, patch `$HOME` mount, install update, launch app |
| `restart` | v0.0.6 | Stop app, delegate to `start` |
| `stop` | v0.0.3 | Stop the running app |
| `logs` | v0.0.3 | Show live app logs |
| `build` | v0.0.2 | Verify libraries, compile sketch, flash via OpenOCD |
| `libs` | v0.0.1 | Library manager — global registry, project assignments, sketch.yaml authority |
| `list` | v0.0.1 | List available apps via `arduino-app-cli` |
| `clean` | v0.0.1 | Full Docker nuke + cache clear + restart |
| `cache` | v0.0.2 | Standalone cache clear (superseded by `start`) |

### 4.2 start (v0.0.6)

The primary command. Every `restart` delegates to `start`.

Sequence:
1. Clear screen, print `=== start ===`
2. Save app name to `~/.last_app`
3. Remove Docker container: `docker rm -f arduino-<app>-main-1`
4. Remove Docker image: `docker rmi -f arduino-<app>-main`
5. Remove `.cache/` directory from app folder
6. Copy `newrepo.bash` from repo to `~/bin/update` and set executable
7. Run `arduino-app-cli app start <app_path>`
8. Wait for `.cache/app-compose.yaml` to be generated (up to 60 seconds)
9. Patch the compose file to mount `$HOME` into the container
10. Restart the container with the patched compose file

**Why the `$HOME` mount patch:** The Docker container generated by `arduino-app-cli` does not mount `$HOME` by default. Python apps that need to persist state (e.g. `~/.scd30-calibrated`) require this mount. The patch inserts a bind mount into the generated compose file and force-recreates the container.

### 4.3 build (v0.0.2)

Compiles and flashes MCU sketches without App Lab.

Sequence:
1. Derive project name from sketch path
2. Run `libs check <project>` pre-flight — abort if any assigned library is not installed
3. Run `arduino-cli compile --fqbn arduino:zephyr:unoq <sketch_path> -v`
4. Run `arduino-cli upload --profile default --fqbn arduino:zephyr:unoq <sketch_path>`

Upload uses OpenOCD via SWD under the hood — the same mechanism App Lab uses, but invoked directly through Arduino CLI.

`sketch.yaml` is owned exclusively by `libs` and is never written or modified by `build`.

### 4.4 libs (v0.0.1)

The library manager. Single source of truth for all Arduino libraries.
Libraries are global to the board. Projects declare which libraries they use.
A library cannot be removed while any project still references it.

`libs` owns `~/.hybx/libraries.json` and is the only command that writes
`sketch.yaml` library sections.

Key subcommands:

```
libs install <n>           -- install globally
libs remove <n>            -- remove (hard blocked if any project uses it)
libs use <project> <n>     -- assign library to project, rewrite sketch.yaml
libs unuse <project> <n>   -- remove assignment, rewrite sketch.yaml
libs sync                     -- rebuild registry from arduino-cli
libs check <project>          -- pre-flight verify used by build
```

All subcommands support `--json` for machine-readable output and `--confirm`
to skip interactive prompts — both required for future GUI integration.

See COMMANDS.md for the full subcommand reference.

### 4.5 Last-App Memory

Commands that accept an optional app name (`stop`, `restart`, `logs`, `clean`) read from and write to `~/.last_app`. If no app name is provided on the command line, the last used app is substituted automatically.

## 5. Bootstrap Script

`scripts/newrepo.bash` is the reference copy of the bootstrap script. It provisions a new UNO Q from scratch.

**Critical rule:** `newrepo.bash` must live in `$HOME` on the UNO Q and is **never** stored directly in the repo root — it must exist before the repo can be cloned. The reference copy in `scripts/` is for documentation and distribution only.

### 5.1 Bootstrap Sequence

1. `cd $HOME && rm -rf Arduino bin Repos` — wipe existing environment
2. `mkdir -p $REPO_DEST && cd $REPO_DEST` — create repo destination
3. `git clone $REPO` — clone UNO-Q robot app repo
4. `git clone $DEV_REPO` — clone UNO-Q-Development-System repo
5. `cp -rp UNO-Q/Arduino $HOME` — deploy apps to `~/Arduino`
6. `cp -rp UNO-Q-Development-System/bin $HOME` — deploy bin commands to `~/bin`
7. Copy `secrets.py.template` to each app that requires it
8. Create versioned symlinks — for each command, find the latest `command-vX.Y.Z.py` and symlink to `~/bin/command`

### 5.2 User-Configurable Variables

Only the top section of `newrepo.bash` needs editing for a new user or board:

```bash
REPO_DEST="$HOME/Repos/GitHub/hybotix"
REPO="https://github.com/hybotix/UNO-Q.git"
DEV_REPO="https://github.com/hybotix/HybX-Development-System.git"
SECRETS_DEST="securesmars"
COMMANDS="board build clean libs list logs project restart setup start stop"
```

### 5.3 update as a Bin Command

After the first `start`, `newrepo.bash` is automatically installed as `~/bin/update` by the `start` command (v0.0.6). From that point forward, re-bootstrapping the environment is simply:

```bash
update
```

## 6. Versioning Conventions

All bin commands follow strict versioning:

- **Filename format:** `command-vX.Y.Z.py`
- **Patch (Z):** bug fixes only, no behavior change
- **Minor (Y):** new features, backward compatible
- **Major (X):** breaking changes
- **Symlinks:** always point to the latest version; updated by `update`
- **Never delete old versions** — they remain in the repo for reference and rollback

When a command is updated:
1. Create the new versioned file (e.g. `start-v0.0.7.py`)
2. Push to the repo
3. Run `update` on the UNO Q to update the symlink

## 7. VSCode Extension

The `uno-q-dev` VSCode extension provides a graphical front-end for all Development System commands. It is developed and maintained alongside this repo.

### 7.1 Architecture

The extension runs on the local machine (Mac/Linux/Windows) and executes commands via the system `ssh` binary — no Node.js SSH library, no Remote-SSH dependency. Any machine that can `ssh arduino@unoq.local` from a terminal can use the extension.

### 7.2 Commands

All commands are available in the VSCode Command Palette under the `UNO Q:` prefix:

| Command | Wraps |
|---------|-------|
| UNO Q: Connect | SSH connection test |
| UNO Q: Start App | `start <app>` with app picker |
| UNO Q: Stop App | `stop` |
| UNO Q: Restart App | `restart <app>` |
| UNO Q: Show Logs | `logs <app>` (streaming) |
| UNO Q: Build Sketch | `build <sketch_path>` |
| UNO Q: Add Library | `addlib <action> <name>` |
| UNO Q: List Apps | `list` |
| UNO Q: Clean | `clean <app>` |
| UNO Q: Run update | `update` |

### 7.3 Status Bar

A status bar item shows the current app state:

- `▶ UNO Q: <app>` — running
- `■ UNO Q: <app> (stopped)` — stopped
- `⬡ UNO Q` — idle

### 7.4 Configuration

| Setting | Default | Description |
|---------|---------|-------------|
| `unoQDev.sshHost` | `arduino@unoq.local` | SSH connection string |
| `unoQDev.appsPath` | `~/Arduino` | Apps directory on UNO Q |
| `unoQDev.sshKeyPath` | *(empty)* | SSH private key path |

### 7.5 Distribution

The extension is distributed as a `.vsix` file. Install via:

```bash
code --install-extension uno-q-dev-0.1.0.vsix
```

Or on Mac if `code` is not in PATH:

```bash
"/Applications/Visual Studio Code.app/Contents/Resources/app/bin/code" \
    --install-extension uno-q-dev-0.1.0.vsix
```

## 8. Known Issues

See [KNOWN_ISSUES.md](KNOWN_ISSUES.md) for full details on open vendor bugs:

- **Docker mDNS resolution** — apps cannot resolve `*.local` hostnames inside Docker containers (Arduino issue [#328](https://github.com/arduino/arduino-app-cli/issues/328))
- **Infineon optigatrust I2C bus** — `liboptigatrust` hardcodes `/dev/i2c-1` (Infineon issue [#26](https://github.com/Infineon/python-optiga-trust/issues/26))

## 9. Key Technical Discoveries

These are undocumented behaviors discovered through reverse engineering the UNO Q platform:

- **QWIIC uses `Wire1`** — the QWIIC connector is on I2C bus 1, not the default bus 0. All QWIIC/Stemma QT sensors must be initialized with `&Wire1`
- **`Bridge.provide()` before `setup()`** — Bridge function registrations must be declared before `setup()` runs or they are not available to the Python side
- **arduino-app-cli without App Lab** — `arduino-app-cli app start` works directly from the command line without the App Lab GUI
- **OpenOCD via SWD** — `arduino-cli upload` uses OpenOCD over SWD internally; this is the same flash mechanism as App Lab
- **`arduino.app_utils` is Docker-injected** — the `arduino.app_utils` Python package is not installable from PyPI; it is injected into the Docker container at runtime by `arduino-app-cli`
